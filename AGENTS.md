# OpenCode Agent 工作流程规范

本文档定义了 OpenCode Agent 执行任务的规范和流程。

---

## 📚 核心文档体系

本项目使用双文档驱动模式，所有任务必须遵循以下文档规范：

### 1. SPEC.md - 技术规范文档

| 属性 | 说明 |
|------|------|
| **位置** | `/root/.openclaw/workspace/test/todo-manager/SPEC.md` |
| **用途** | 定义技术架构、接口规范、数据模型、编码标准 |
| **何时使用** | **任务执行前**必须阅读，确保实现符合技术规范 |
| **更新时机** | 技术架构变更、新增接口、修改数据模型时 |
| **核心内容** | 系统架构图、接口定义、数据模型、异常处理、性能指标 |

**Agent 职责**:
- 实现功能前，检查 SPEC 中的相关规范
- 确保代码符合架构设计要求
- 接口实现必须遵循 SPEC 定义

---

### 2. PLAN.md - 项目计划文档

| 属性 | 说明 |
|------|------|
| **位置** | `/root/.openclaw/workspace/test/todo-manager/PLAN.md` |
| **用途** | 定义里程碑、任务分解、进度跟踪、验收标准 |
| **何时使用** | **任务执行前后**必须更新，跟踪任务状态 |
| **更新时机** | 每个任务完成后立即更新进度 |
| **核心内容** | 里程碑规划、任务队列、依赖关系、完成状态 |

**Agent 职责**:
- 任务开始前，从 PLAN 中获取当前优先级任务
- 任务完成后，立即更新 PLAN 中的任务状态
- 将「⏳ 待开始」改为「✅ 已完成」
- 如产生新任务，添加到 PLAN 任务队列

---

## 🎯 基本原则

1. **文档驱动** - 所有工作基于 SPEC 和 PLAN 双文档
2. **SPEC 为纲** - 技术实现必须符合 SPEC 规范
3. **PLAN 为领** - 任务执行必须跟随 PLAN 规划
4. **非必要不更新 PLAN** - 只有真正完成任务后才更新 PLAN 状态，禁止虚假更新
5. **及时更新** - 每完成一个任务立即更新 PLAN
6. **一次只执行一个任务** - 专注于当前任务，不发散
7. **测试驱动** - 每个功能必须有测试验证
8. **持续集成** - 每个任务完成后立即提交

---

## 📋 工作流程（六阶段）

### 阶段 0: 文档准备（新增）

**执行步骤**:
1. **阅读 SPEC.md** - 了解技术规范、接口定义、架构设计
2. **阅读 PLAN.md** - 了解当前里程碑、待完成任务
3. 确认当前最高优先级任务

**输出**:
- 技术实现方案（基于 SPEC）
- 当前执行任务（来自 PLAN）

---

### 阶段 1: 任务分析

**执行步骤**:
1. 读取任务描述（从 PLAN）
2. 查阅 SPEC 中的相关规范
3. 分析需求和技术方案
4. 检查现有代码结构
5. 确定实现策略

**约束**:
- 实现必须符合 SPEC 规范
- 接口设计必须符合 SPEC 定义
- 数据模型必须符合 SPEC 定义

**输出**:
- 任务理解摘要
- 技术方案概要（引用 SPEC 章节）
- 可能遇到的问题预测

---

### 阶段 2: 代码实现

**执行步骤**:
1. 编写/修改代码
2. **遵循 SPEC 中的编码规范**
3. 添加必要的注释
4. 确保代码可读性
5. **检查是否符合架构设计（SPEC）**

**约束**:
- 不要修改与当前任务无关的代码
- 保持向后兼容性
- 使用项目已有的技术栈
- **接口必须符合 SPEC 定义**

---

### 阶段 3: 功能测试

**执行步骤**:
1. 运行单元测试（如果有）
2. 手动验证功能
3. 检查边界情况
4. 验证错误处理
5. **对照 SPEC 验收标准检查**

**通过标准**:
- [ ] 功能按预期工作
- [ ] 无明显 bug
- [ ] 错误提示清晰
- [ ] **符合 SPEC 中的验收标准**

---

### 阶段 4: 文档更新

**执行步骤**:
1. 如技术架构变更，**更新 SPEC.md**
2. 如新增接口，**更新 SPEC.md 接口章节**
3. 更新 README（如需要）
4. 添加/更新代码注释

**约束**:
- SPEC 是技术规范的权威来源
- 代码与 SPEC 必须保持一致

---

### 阶段 5: Git 提交

**执行步骤**:
1. `git add -A`
2. `git commit -m "类型: 简短描述"`
3. `git push origin main`

**提交信息格式**:
```
类型: 简短描述

- 详细变更点 1
- 详细变更点 2
- 关联 SPEC 章节: X.X
- 完成 PLAN 任务: X.X
```

类型包括:
- `Feat`: 新功能
- `Fix`: 修复
- `Refactor`: 重构
- `Docs`: 文档
- `Test`: 测试

---

### 阶段 6: 任务更新（关键）

**核心原则：非必要不更新 PLAN**

只有在以下**全部条件**满足时才更新 PLAN：
- ✅ 代码已实际编写并通过测试
- ✅ 功能已验证可用
- ✅ Git 已提交
- ✅ 不是虚假/形式上的完成

**执行步骤**:
1. **验证完成质量** - 确认任务真正完成，不是虚假更新
2. **更新 PLAN.md** - 将完成的任务标记为「✅ 已完成」
3. 如产生新任务，添加到 PLAN 任务队列
4. 更新任务进度百分比

**禁止**:
- ❌ 未实际完成就标记为 ✅ 已完成
- ❌ 仅更新 PLAN 状态而不完成实际工作
- ❌ 形式上的完成（如 OpenCode 未实际执行成功）
4. 如有阻塞问题，在 PLAN 中记录

**PLAN 更新格式**:
```markdown
| 优先级 | 任务 | 状态 | 预计工时 | 实际工时 |
|--------|------|------|----------|----------|
| 🔴 高 | XXX | ✅ 已完成 | 2h | 1.5h |
```

**重要**: 每个任务完成后**必须**更新 PLAN.md，这是强制要求。

## 🚫 禁止事项

- ❌ 不阅读 SPEC 直接开始编码
- ❌ 不更新 PLAN 就认为是任务完成
- ❌ **虚假更新 PLAN** - 未实际完成任务就标记为 ✅ 已完成
- ❌ **形式完成** - 仅更新状态而不完成实际工作
- ❌ 一次修改多个不相关的功能
- ❌ 提交未经测试的代码
- ❌ 提交带有敏感信息的代码
- ❌ 破坏现有功能
- ❌ 过度工程化

## 📝 示例工作流

```
任务: 添加任务优先级功能

0. 文档准备
   - 阅读 SPEC.md 第 3 章（数据模型）
   - 阅读 SPEC.md 第 4 章（CLI 接口）
   - 阅读 PLAN.md M1 任务列表

1. 分析
   - 需要修改 Task 模型（按 SPEC 3.1）
   - 需要添加 set-priority 命令（按 SPEC 4.1）
   - 优先级枚举: HIGH/MEDIUM/LOW（按 SPEC 3.2）

2. 实现
   - 修改 core/models.py 添加 priority 字段
   - 创建 commands/set_priority.py
   - 实现优先级修改逻辑

3. 测试
   - 运行 pytest tests/unit/commands/test_set_priority.py
   - 手动验证: todo set-priority 1 高
   - 检查边界: 无效优先级处理

4. 文档
   - 更新 SPEC.md（如接口有变更）
   - 代码添加 docstring

5. 提交
   - git add .
   - git commit -m "Feat: 添加任务优先级功能

     - 在 Task 模型添加 priority 字段
     - 添加 set-priority CLI 命令
     - 关联 SPEC: 3.1, 4.1
     - 完成 PLAN: M1.3"
   - git push

6. 更新 PLAN
   - 打开 PLAN.md
   - 找到 "添加任务优先级功能" 行
   - 修改状态: ⏳ 待开始 → ✅ 已完成
   - 更新实际工时: 2h
   - git add PLAN.md && git commit -m "Docs: 更新 PLAN 进度"
   - git push
```

## 🔄 反馈循环

如果任务执行失败:
1. 分析失败原因
2. 修复问题
3. 重新测试
4. 再次提交
5. **更新 PLAN 记录问题和解决方案**
6. 记录教训
