# OpenCode Agent 工作流程规范

本文档定义了 OpenCode Agent 执行任务的规范和流程。

**版本**: 3.0 - BabyAGI Executor 架构  
**更新**: 2026-03-04

---

## 🏗️ 架构概述

本项目采用 **BabyAGI Executor** 架构：

```
┌─────────────────────────────────────────┐
│         BabyAGI Executor                │
│    (Python 自动化任务执行引擎)           │
└─────────────────────────────────────────┘
                    │
        ┌───────────┼───────────┐
        │           │           │
        ▼           ▼           ▼
   ┌────────┐  ┌────────┐  ┌────────┐
   │  Task  │  │ Prompt │  │ Memory │
   │ Queue  │──│ Engine │──│ Store  │
   │(JSON)  │  │(OpenCo)│  │(Files) │
   └────────┘  └────────┘  └────────┘
                    │
                    ▼
            ┌───────────────┐
            │  OpenCode CLI │
            │(serve+attach) │
            └───────────────┘
```

**核心变化** (相比 2.0):
- 任务存储: `PLAN.md` → `executor/tasks/tasks.json` (JSON 格式)
- 执行模式: 直接调用 → `serve + attach` 模式
- 结果解析: 文本推断 → XML 结构化解析
- 输出格式: 自由文本 → 强制 XML 格式

### 1. AGENTS.md - 工作流程规范

| 属性 | 说明 |
|------|------|
| **位置** | `./AGENTS.md` |
| **用途** | 定义 OpenCode Agent 工作流程、规范、检查清单 |
| **何时使用** | **每次任务执行前**必须阅读 |
| **关键内容** | 六阶段工作流、XML 输出格式、Git 提交规范 |

### 2. SPEC.md - 技术规范文档

| 属性 | 说明 |
|------|------|
| **位置** | `./docs/SPEC.md` |
| **用途** | 定义技术架构、接口规范、数据模型、编码标准 |
| **何时使用** | **任务执行前**必须阅读 |
| **更新时机** | 技术架构变更时 |

### 3. tasks.json - 任务数据

| 属性 | 说明 |
|------|------|
| **位置** | `./executor/tasks/tasks.json` |
| **用途** | **执行器任务队列数据源**，JSON 格式 |
| **格式** | 结构化任务列表，包含 id, objective, type, priority, status 等 |
| **何时使用** | 执行器自动读取，Agent 无需直接修改 |
| **更新时机** | 任务执行完成后由执行器自动更新 |

**任务数据结构**:
```json
{
  "id": "task-001",
  "objective": "实现功能",
  "type": "coder",
  "priority": "🔴",
  "status": "pending",
  "estimated_hours": 1.0,
  "tags": ["feature"]
}
```

### 4. Thinking 文档 - 思考记录

| 属性 | 说明 |
|------|------|
| **位置** | `./executor/thinking/task-{ID}-{名称}.md` |
| **用途** | 记录 AI 思考过程、分析结论、决策依据 |
| **何时使用** | **阶段 0 加载最近 2 次**，阶段 6 保存当前 |
| **更新时机** | 每个任务完成后由执行器自动创建 |

**Thinking 文档加载规则**:
- 执行器自动加载最近 **2 个** Thinking 文件到 OpenCode 上下文
- 帮助 AI 了解前序任务的思路和结论
- 避免重复思考，保持思路连续性

### 5. PLAN.md - 项目计划（参考）

| 属性 | 说明 |
|------|------|
| **位置** | `./docs/PLAN.md` |
| **用途** | 里程碑规划、高层任务分解（**仅作参考**） |
| **注意** | **执行器不再从 PLAN.md 读取任务**，改为 tasks.json |
| **更新时机** | 手动维护，记录高层规划 |

## 📝 任务管理规范

### 任务数据源

**重要**: 执行器从 `executor/tasks/tasks.json` 读取任务，**不再解析 PLAN.md**。

如需添加新任务：
1. 直接编辑 `executor/tasks/tasks.json`
2. 或使用执行器的任务管理 API

### 优先级定义

| 优先级 | 标识 | 使用场景 | 示例 |
|--------|------|----------|------|
| **紧急** | 🔴🔴 | 阻塞性问题、严重 bug | 修复崩溃问题 |
| **高** | 🔴 | 核心功能、必须完成 | 实现核心 API |
| **中** | 🟡 | 重要但可延期 | 性能优化 |
| **低** | 🟢 | 锦上添花 | 代码美化 |

### 任务类型

| 类型 | 说明 | Prompt 模板 |
|------|------|-------------|
| **coder** | 代码开发任务 | `prompts/coder.txt` |
| **repair** | 修复任务 | `prompts/repair.txt` |
| **planner** | 任务分解 | `prompts/planner.txt` |
| **docs** | 文档任务 | 使用 coder 模板 |

### 工时估算

| 规模 | 工时 | 说明 |
|------|------|------|
| 小任务 | 0.5h | 简单修复、单文件修改 |
| 中任务 | 1-2h | 功能实现、多文件修改 |
| 大任务 | 3h+ | 复杂功能、架构调整 |

**建议**: 宁可低估工时，不要高估。可以拆分大任务。

## 📋 工作流程（六阶段）

### 阶段 0: 文档准备

**执行步骤**:
1. **阅读 AGENTS.md** - 了解工作流程规范（**必读**）
2. **阅读 SPEC.md** - 了解技术规范、接口定义、架构设计
3. **加载 Thinking 文档** - 读取最近 2 个任务的思考记录
   - 路径: `executor/thinking/task-*.md`
   - 目的: 了解前序任务的思路、结论、踩过的坑
   - 加载数量: 最近 2 个（避免上下文过长）
4. 确认当前最高优先级任务（由执行器提供）

**输出**:
- 工作流程理解（来自 AGENTS）
- 技术实现方案（基于 SPEC）
- 历史思路参考（来自 Thinking）

---

### 阶段 1: 任务分析

**执行步骤**:
1. 读取任务描述（由执行器注入 Prompt）
2. 查阅 SPEC 中的相关规范
3. 分析需求和技术方案
4. 检查现有代码结构
5. 确定实现策略

**约束**:
- 实现必须符合 SPEC 规范
- 接口设计必须符合 SPEC 定义
- 数据模型必须符合 SPEC 定义

**输出**:
- 任务理解摘要
- 技术方案概要（引用 SPEC 章节）
- 可能遇到的问题预测

---

### 阶段 2: 代码实现

**执行步骤**:
1. 编写/修改代码
2. **遵循 SPEC 中的编码规范**
3. 添加必要的注释
4. 确保代码可读性
5. **检查是否符合架构设计（SPEC）**

**约束**:
- 不要修改与当前任务无关的代码
- 保持向后兼容性
- 使用项目已有的技术栈
- **接口必须符合 SPEC 定义**

---

### 阶段 3: 功能测试

**执行步骤**:
1. 运行单元测试（如果有）
2. 手动验证功能
3. 检查边界情况
4. 验证错误处理
5. **对照 SPEC 验收标准检查**

**通过标准**:
- [ ] 功能按预期工作
- [ ] 无明显 bug
- [ ] 错误提示清晰
- [ ] **符合 SPEC 中的验收标准**

---

### 阶段 4: 文档更新

**执行步骤**:
1. 如技术架构变更，**更新 SPEC.md**
2. 如新增接口，**更新 SPEC.md 接口章节**
3. 更新 README（如需要）
4. 添加/更新代码注释

**约束**:
- SPEC 是技术规范的权威来源
- 代码与 SPEC 必须保持一致

---

### 阶段 5: Git 提交

**执行步骤**:
1. `git add -A`
2. `git commit -m "类型: 简短描述"`
3. `git push origin main`

**提交信息格式**:
```
类型: 简短描述

- 详细变更点 1
- 详细变更点 2
- 关联 SPEC 章节: X.X
- 完成任务: task-XXX
```

类型包括:
- `Feat`: 新功能
- `Fix`: 修复
- `Refactor`: 重构
- `Docs`: 文档
- `Test`: 测试

---

### 阶段 6: 输出结果报告（关键）

**核心原则**: 必须输出 XML 格式结果，执行器依赖此格式解析

**输出格式** (必须严格遵循):

```xml
<result>
  <status>success|failed</status>
  <summary>任务完成情况的简短总结（100字以内）</summary>
  <files_changed>
    <file>修改或创建的文件路径1</file>
    <file>修改或创建的文件路径2</file>
  </files_changed>
  <tests>
    <passed>true|false</passed>
    <details>测试结果详情</details>
  </tests>
  <git>
    <committed>true|false</committed>
    <commit_hash>提交的 hash（如果有）</commit_hash>
  </git>
  <notes>其他需要说明的信息</notes>
</result>
```

**重要**:
- 必须包含完整的 XML 格式结果
- `status` 必须是 `success` 或 `failed`
- 所有修改的文件必须列在 `files_changed` 中
- 对于 repair 类型，需额外包含 `<root_cause>` 字段

**执行器会自动**:
- 解析 XML 结果
- 更新 tasks.json 中的任务状态
- 创建 Thinking 文档保存输出内容
- 记录成功/失败经验到长期记忆

## 🔧 执行器架构

### 核心组件

| 组件 | 文件 | 职责 |
|------|------|------|
| **TaskManager** | `tasks/manager.py` | 管理 tasks.json 的 CRUD |
| **TaskQueue** | `core/queue.py` | 任务队列，基于 TaskManager |
| **ExecutionRunner** | `core/runner.py` | 主执行引擎，BabyAGI 循环 |
| **OpenCodeClient** | `opencode/client.py` | OpenCode serve+attach 调用 |
| **ResultParser** | `opencode/parser.py` | 解析 XML 输出 |
| **Memory** | `memory/*.py` | 短期/长期记忆管理 |

### 执行模式

**serve + attach 模式**:
1. 首次执行时启动 `opencode serve --port 4096`
2. 后续任务使用 `opencode run --attach http://localhost:4096`
3. 避免 MCP 冷启动，提高执行效率
